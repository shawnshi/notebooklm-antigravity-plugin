## 2024-06-22 - [JSON extraction bottleneck]\n**Learning:** The use of `re.search(r'(\{.*\}|\[.*\])', text, re.DOTALL)` to extract JSON from text containing large amounts of noise (like Chromium logs) can be extremely slow due to regex backtracking and copying over large strings. String `.find()` and `.rfind()` methods are order of magnitude faster.\n**Action:** Replace regex-based JSON extraction with simple string `.find()` and `.rfind()` where possible, especially on texts containing arbitrary prefixes/suffixes.
## 2024-05-15 - Unused standard library imports impact startup time
**Learning:** In short-lived CLI wrapper architectures where Python scripts execute from scratch for every event, standard library imports like `re` add ~15-20ms of overhead. Even if documentation mentions regular expressions, native string methods are often used instead.
**Action:** Always verify if expensive standard library imports are actually used in short-lived bridge scripts, as removing them provides a measurable reduction in startup time.
## 2024-07-06 - [Redundant text.strip() allocation before json.loads()]
**Learning:** `json.loads()` natively ignores leading and trailing whitespaces. Calling `.strip()` on a potentially large string payload (like a 10MB chunk from a standard output containing Base64 data or LLM payloads) before JSON parsing creates an entirely unnecessary string copy in memory, causing substantial CPU overhead.
**Action:** Remove `.strip()` before calling `json.loads()` or `text.find()`. Use the string directly to prevent unnecessary string copies and O(N) allocation overhead.
## 2024-07-06 - [Redundant JSON re-serialization overhead]
**Learning:** Extracting JSON via `json.loads()` and then immediately re-serializing it using `json.dumps()` creates significant CPU and memory overhead, especially for large payloads like 10MB reports. Validating JSON by parsing is necessary, but the re-serialization step is completely redundant when the raw text was already valid JSON.
**Action:** When acting as a bridge or wrapper that just passes JSON through, parse it for validation but return/print the original matched raw string directly instead of re-serializing it with `json.dumps()`.
## 2024-07-28 - [O(1) JSON Fallback Extraction]
**Learning:** Extracting embedded JSON from large strings using `text[start:end]` and `json.loads()` creates O(N) memory allocations due to string slicing. In cases with large prefixes/suffixes or extremely large invalid strings, this can cause significant memory spikes and slowdowns. Using `json.JSONDecoder().raw_decode(text, start_idx)` evaluates and parses the JSON starting at `start_idx` in-place, reducing memory overhead to O(1) auxiliary space (excluding the parsed object itself). Also, it inherently avoids the logic bugs associated with blindly matching the first `{` and last `}` which fails when multiple JSON payloads exist in the same string.
**Action:** When extracting JSON from a string containing noise, use `json.JSONDecoder().raw_decode(text, start_idx)` instead of manual string slicing and `json.loads()`.
